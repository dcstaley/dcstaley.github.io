Unified Tasks & Habits App

A local-first web app that puts tasks, habits, and calendar events on one daily view. A 9:00am meeting from Google Calendar, a by-end-of-day errand, and a four-times-a-week habit sit side by side, each scheduled as loosely or as precisely as it needs.

v1 shipped · three phases Sole product owner 2026–present Next.js · PowerSync/SQLite · Supabase
Try it live (desktop only) Read the work samples See the v1 feature set

A live, evolving product, actively developed, with v2 underway.

The problem

Planning a real day means looking in three places: a task app, a habit tracker, and a calendar. Each has its own model, and each demands more precision than real plans have. A to-do app wants a date, a calendar wants a time, and neither has a home for "practice Spanish four times this week." The product's bet is that these are the same thing: one item model on one daily view, with your real calendar alongside it.

Discovery: the incumbent

Before I wrote the spec, I looked closely at TickTick, the best-known app that already combines tasks, habits, and a calendar. It has a day timeline, a rail of unscheduled tasks you drag onto it, durations, and recurring tasks. Its scheduling works in two modes: an item has a specific time, or it has none. There's no flexible option in between, no way to tell it "I'll do this sometime between 12 and 4" and have it sit on the day without committing to a slot. Habits are separate as well, tracked in their own module and shown on the calendar rather than planned like a task. That's the space I built for: flexible scheduling, and habits that work like any other item.

Competitive analysis · TickTick, structural insight

"TickTick unified the apps but not the model. Its architecture keeps task, habit, and calendar event as different kinds of things, then works to display them together... The real opening is not another feature-complete competitor. It is a product built on one item model: task, event, and habit as the same underlying item with options turned on, rendered on one daily view next to the user's real calendar."

How I run it

Like a product organization of one. I maintain a living PRD, a Jira backlog of stories with acceptance criteria, and narrative specs for the information and technical architecture. AI builds each piece from the written spec, and I review it, request changes, and accept it only once it meets the acceptance criteria I set.

Key decisions

One item type, or tasks vs. habits?

One type. Repeat and habit tracking are optional toggles on the same item, so a weekly meeting, an errand, and a four-times-a-week habit all live in one create flow and one daily view. This is the product's core bet, and it fills the gap the incumbents leave open.

Require an account?

No. The app is local-first and fully usable as a guest; an account adds sync and backup rather than gating access. It also inverts the incumbent's paywall on the features that define the category.

Past days: read-only?

I reversed my own call. Read-only was my original stance. It punished anyone who simply forgot to check an item off, so completions are now editable for past dates. The reversal is recorded in the PRD changelog with the reasoning.

How precise does scheduling have to be?

As precise or as loose as the task is. Alongside a fixed time, an item can be a flexible window, something due by end of day, or an untimed to-do. All of them sit on the same day view and side rail, not only the items that map to a clock time.

Roster Optimizer

An analytics web app for Out of the Park Baseball's Perfect Team mode. It imports player data, scores every player through regression models trained on real game outcomes, and solves for the optimal 26-player tournament roster under budget, position, and eligibility constraints.

Live and in active development Sole product owner 2025–present TypeScript · React · HiGHS LP solver
Read the work samples See the feature set

The problem

Building a tournament roster is a constrained optimization problem: thousands of players, budget caps, position eligibility, platoon matchups, tier limits. Most people solve it by intuition. I wanted rosters chosen on evidence, with player value predicted by models trained on actual game outcomes rather than player ratings taken at face value. A first version worked, then ran into a problem that reframed the whole project.

Key decisions

V1 is misbehaving. Patch or rebuild?

Rebuild, once discovery justified it. Scores shown in the interface had diverged from the scores the optimizer used, so the same player was valued two ways. Before committing either way, I ran a file-by-file audit that traced every defect to its root cause. The evidence said the architecture could not be trusted, so I rebuilt against a written set of requirements and decisions.

How do you stop scoring drift for good?

Structurally. One scoring core, computed once, used identically by the data grid, the optimizer, and model validation. One language, one shared module. A scoring bug can now exist in only one place.

Which prediction model?

Let real data pick. Rather than choose a model up front, I made it a swappable component behind one interface and built a comparison harness. A bake-off on season results from five simulated leagues decided it, and the winning model is deployed.

Keep matching the old app's scores?

No. Reproducing v1's scores was the rebuild's original success metric. Once the data-driven model outperformed them, that target stopped being useful, so I retired the benchmark and replaced it with model accuracy against real outcomes.

From the actual documents

One decision captures the approach: choosing not to decide yet. Rather than guess the prediction model up front, I made it swappable and let a bake-off on real season data pick it, so the choice was cheap to defer and cheap to change. The document records that reasoning.

Requirements & decisions doc · D3, model functional form

"Defer the functional-form choice to a dedicated data-exploration phase. The model is a swappable component behind a clean interface, so the form can change later without touching anything downstream. A model-comparison harness fits candidate forms, scores them against the diagnostic suite on real data, and selects."

Read the decision log →

Execution

A milestone roadmap with user stories and exploration spikes gating the risky steps: scoring core first, then the data and config layer, the data grid, the optimizer, manual drag-and-drop overrides, and model training. Each milestone was validated before the next began. Recent features come from my own use in live play, such as Biggest Upgrades, which surfaces the acquisition targets that would most improve the current roster.

What this demonstrates

  • A file-by-file audit. Traced the failure to scores computed two different ways, which pointed at the architecture rather than a missing feature and set the scope for a rebuild.
  • A written decision log, D1 through D7. Every call recorded with its reasoning, including the deliberate choice to defer the model until the data existed.
  • A retired success metric. When the data-driven model beat the old app's scores, I dropped parity as the goal and replaced it with accuracy against real outcomes.
  • A full spec set. Discovery findings, requirements, a decision log, and an architecture doc, enough to run the whole rebuild from the documents.

How I work

The two case studies above are products I built end to end; these are the principles behind that work.

Evidence over assumptionI audit what a system actually does, and study the products already out there, before deciding what to build.
Decisions in writingI record settled decisions and the reasoning behind them, so they can be referenced later instead of reopened.
Scope is the productI keep an explicit out-of-scope list and a phased roadmap, and I would rather cut a feature early than late.
Change the call when the evidence doesWhen new data or real use contradicts a decision I've made, I change it and record why, rather than defend it.
