Roadmap and Backlog: OOTP Roster Optimizer Rebuild

Work sample · roadmap, milestones, and risk-gating spikes · as of v1 (July 2026)

What this is: the living plan that turned the settled decisions (D1–D7) into milestones, user stories, and exploration spikes to build against.

What it shows: roadmapping and sequencing, dependency management, and risk-gating. It uses spikes to de-risk the uncertain pieces before committing a milestone, and it keeps a running list of forward risks and suspected issues to revisit.

Source: condensed and edited from the project's roadmap document for presentation (trimmed for length; per-story acceptance criteria, build logs, and internal references removed). The full working version lives in the app repo.

The product, in one line: a single-user desktop-style web app (local server plus browser) that imports card data, scores every card through one shared core, and builds optimal rosters and lineups under tournament rules, with model training and a Single-Player mode.

The through-line: every milestone reads its scores from a single core and never recomputes them, so the numbers can't drift between screens.

The strategy: parity first, then upgrade. Reproduce the old app's trusted numbers exactly, prove it, then improve on that trusted base. Parity was achieved (the rebuilt core reproduces the old app's scores bit-for-bit on the same inputs), after which the project moved to the model upgrade, the D3 bake-off, which selected the raw-poly form that is now the deployed scoring model.

Milestone map

# Milestone Goal Status

M1 Scoring core One core; reproduce the old app's scores exactly Done
M1.5 Self-contained calibration Compute our own anchor and calibration scales Done
M2 Data layer and config Catalog, per-account overlays, Tournament/Era/Park libraries Done
M3 Data Grid First UI consumer of the core Done
M4 Optimizer Roster, lineups, rotation, and bullpen Done
M5 Manual editing Drag-drop roster and lineup overrides Done
M6 Training and bake-off Fit models; run the D3 comparison harness Done (D3 resolved to raw-poly, now the deployed model)
M7 Single Player Import adapter plus potential ratings Planned
X Cross-cutting Packaging, persistence, parity method Ongoing

Sequencing rationale: M1.5 finishes the scoring story, so scales are computed rather than pasted from the old app. M2 builds the structure everything else reads. M3 makes it visible and checkable. M4 is the payoff. M5 through M7 round it out. Spikes gate the milestones that need exploration first.

Each milestone carried a parity gate where an oracle existed: the rebuilt piece had to match the old app on the same inputs before the work was considered done. Past scoring the gate loosens by design, since rosters are "comparable" rather than identical because the decisions deliberately changed optimizer behavior, so M4 needed agreed acceptance criteria so that intended improvements would not be read as regressions.

Spikes: de-risking before committing

The riskiest milestones were gated by a spike that answered one concrete question before any scope was committed. Each spike stated the question, why it mattered, a rough approach, and the output it had to produce.

Spike Question Unblocks
SP-1 How does the old app build its calibration anchor pool and per-event scales? M1.5
SP-2 What is the file-based persistence shape? M2
SP-3 Variant boost and eligibility exact behavior M2
SP-4 Optimizer LP formulation in an in-process WASM solver M4
SP-5 Defensive feasibility as a matching/assignment problem M4
SP-6 Solver performance on real pool sizes M4
SP-7 Does the signed-distance value give a sane hitter/pitcher balance? M4
SP-8 The D3 model functional-form bake-off M6
SP-9 Training-data loader and parity against the old trainer M6
SP-10 Single-Player export format and potential-rating fields M7
SP-11 Packaging scaffold (local server plus browser) M3

The discipline paid off where it mattered most, the optimizer. SP-6 proved the in-process WASM solver handled the real pool fast enough (the full undecomposed problem solved to optimal in about half a second, with the planned decomposition giving a large additional margin), which retired the risk of needing a heavier solver. SP-4 and SP-5 proved the roster problem could be expressed as a single assignment MILP, including expressing the backup catcher as a coverage-depth constraint, which is what let the decisions retire the old app's three-pass catcher workaround. SP-7 measured the hitter/pitcher balance under the new value definition and surfaced a real skew, which led to a tunable per-tournament emphasis knob as the interim control and a baseline recentering as the proper fix. In each case the uncertain thing was tested in isolation before the milestone committed to it.

Open questions and risks

Kept as a live list so nothing quietly slips:

  • The optimizer is the hardest part, so it was de-risked early with spikes before M4 scope was committed.
  • Parity loosens past scoring. Scores are exact; rosters are comparable, so M4 needs explicit acceptance criteria so intended improvements are not mistaken for regressions.
  • Single-source config is a standing invariant, not a one-time task. The old app's split config and duplicated homes must not re-emerge, so the tournament-as-single-source rule is enforced continuously.
  • Calibration and scaling correctness was carried into M6 and evaluated there against real outcome data. The suspected issues (which components are independently re-anchored, how the platoon split is weighted) were decoupled from the optimizer, which reads scoring values as coefficients, so recalibration changed the numbers, not the machinery.

Parity as a baseline, not an endorsement

A standing policy governed how the old app was treated: it is usable but may contain real bugs. It was reproduced for a safe parity baseline, but scrutinized while porting, with suspected bugs flagged rather than copied, and fixes deferred until after parity unless trivial and agreed. Reproducing a number was never an endorsement of it. Two representative finds:

  • A pitch-type counting bug. The same logic lived in two places with two different column-name spellings; one looked for names that never matched, so two pitch types were never counted, quietly affecting starter qualification.
  • A calibration-skewing bug. The pool used to anchor calibration was gated on a position-eligibility flag that wrongly excluded some players, skewing the very scales that drive every displayed score. The correct design anchors against the full eligible pool.

This is the habit the rebuild ran on: reproduce faithfully, verify against evidence, and surface what looks wrong rather than assuming the prior version was right.
