Discovery Findings: OOTP Roster Optimizer Rebuild

Work sample · codebase discovery and root-cause analysis · as of v1 (July 2026)

What this is: the discovery pass over the existing app's codebase, done before committing to a rebuild. Ground truth is the code itself, cross-checked against the earlier handover document.

What it shows: evidence-based discovery and root-cause analysis. It inventories what every file actually does, separates live code from dead, and corrects the places where the handover's assumptions turned out to be wrong.

Source: condensed and edited from the project's discovery-findings document for presentation (trimmed for length, internal references removed). The full working version, with the complete file-by-file inventory, lives in the app repo.

The short version. The rebuild's founding premise was "one scoring core, computed once, consumed identically everywhere." This pass tested that premise against the real code. The finding: the per-card formulas were already consolidated into one module, so that was not the problem. The genuine problem was the calibration and scaling layer, which was re-implemented up to four times and split across the frontend and backend. Several handover claims were also disproved by the code. These findings are what shaped the requirements and decisions that followed.

A note on method: live-versus-dead status is derived from the import graph, which is reliable. Small and medium files were read in full; the few very large files (a roster generator near 4,800 lines, the roster page near 3,000, the server near 2,000) were read in their relevant sections and characterized structurally.

1. Headline findings (the things that change the rebuild picture)

1.1 The handover had the scoring core backwards. It called one file the authoritative path and said the actual core was deleted. The reverse is true: the core module is alive and is the single formula core; the file the handover named is a thin per-row wrapper that delegates every formula to it. Good news underneath the error: the formula bodies really are consolidated into one module on the frontend.

1.2 The real "scoring in multiple places" problem is the frontend/backend split, plus calibration scatter. There are two copies of the formula core, one frontend and one backend, kept in sync by hand and currently identical, which is a latent drift source. The backend copy is imported by nothing; the optimizer re-derives scoring inline rather than calling the core. So the formulas are consolidated on the frontend, but the backend optimizer re-derives scores independently, and the backend keeps an orphaned synced copy that nothing uses. Calibration and scaling is the genuinely scattered layer, appearing across at least nine files. The divergence is in the code explicitly: a comment notes a cross-pool pitcher multiplier that is "LP-only, not for display," so the same player can be scored differently for the grid versus the optimizer.

1.3 Persistence is a hybrid, not "all IndexedDB" as documented. IndexedDB holds the dataset, variants, eras, and tournaments; localStorage holds coefficients, grid state, and calibration scales. Two declared IndexedDB stores are never used (those values live in localStorage instead), leaving orphaned store definitions.

1.4 "No default coefficients" is false. The docs said a trained model supplies all coefficients. In fact the app ships a complete default coefficient set with real fitted values baked in, and scores against these on first load before any model is loaded. Whether that is desirable is a rebuild decision, but it is not "no defaults."

1.5 There are remnants of an older, all-frontend solver architecture. The app migrated from an in-browser JavaScript solver to a backend Python/HiGHS solver, and several leftover files (a frontend LP optimizer, a client for an endpoint that no longer exists, a browser-script loader, an unmounted Express router) are now dead code, imported by nothing.

1.6 The live solve path. The solver wrapper writes the LP to a temp file and shells out to a Python highspy process. So the Python file is not an "alternative" path as the handover implied; it is the solver.

1.7 Files the handover never mentioned. Several live, shared modules and UI components were absent from the handover entirely, including the shared position-constraints module (which matters, because the handover claimed constraints were frontend-only), the localStorage helpers, and a Python recalibration script at the repo root.

1.8 Smaller corrections. The era record has no HR toggle (the toggle lives in coefficients, not the era); position constraints are split, with definitions in page state but enforcement in a shared backend module; the Single Player page is literally named and routed as "Scouting," so the misnomer the handover wanted renamed is baked into filenames and routes.

2. File inventory (what is live, dead, and orphaned)

The full pass read or structurally characterized every non-vendored source file. The classification that matters for the rebuild:

  • The formula core is one frontend module of pure functions (softcap, park compression, basic and advanced hitting/pitching, wOBA). It is authoritative, contrary to the handover.
  • The backend optimizer builds the LP and runs multi-phase post-processing, and re-derives scoring inline from a modifiers bundle the frontend sends rather than calling a shared core. This is the central architectural issue.
  • Orphaned or dead: the backend's hand-synced copy of the core (imported by nothing), a dead frontend LP optimizer, a client for a non-existent endpoint, a browser-script solver loader, an unmounted Express router, legacy seed models, solver debug dumps, and two unused IndexedDB store declarations. All are safe-to-drop candidates.
  • Live but undocumented: the shared position-constraints module, the localStorage helpers, the IndexedDB helpers, and several editing widgets (rotation, depth chart, settings, defensive input).

The takeaway is not the file list itself but its shape: a consolidated formula core, a backend that bypasses it, a scattered calibration layer, and a visible layer of dead code from a prior architecture.

3. The three flagged open problems, confirmed against the code

Scoring, scaling, and calibration layering (highest value). Confirmed as the real mess. The per-card formulas are consolidated; the calibration and scaling layer is split frontend to backend via the modifiers bundle, re-derived in the LP rather than consumed, and already divergent between display and optimizer. The distinct operations to untangle, as they exist today: per-event softcaps, anchor calibration, calibration scales cached for display, and a cross-pool pitcher/hitter multiplier applied only in the optimizer. Whether all of these are necessary, and their order versus era and park, is the design question, and now the exact code for each can be pointed at.

Model functional form. Confirmed: the coefficient store carries unused quadratic and cubic slots (all zero) alongside the log terms, exactly the dead slots that make the mixed form nonsensical. Re-deriving the form against residuals is open.

Multi-source config split. Confirmed, and slightly worse than described: generation config is split across the IndexedDB tournament and the page-level roster settings, and position-constraint enforcement lives in a separate backend module. Park factors and softcaps each duplicate between the tournament and the coefficients store.

4. Calibration layer, end to end (the deep dive)

4.1 The wOBA-to-calibrated-score pipeline is re-implemented four times. The per-event to BIP to hits to wOBA recomputation, with calibration scales, era, and park, exists nearly identically in four places: the core (raw, no era/park/calibration), the roster page (frontend, calibrated), the backend optimizer (the same steps again, to build LP coefficients), and the backend anchor computation (the same recompute a fourth time, to derive the calibration scales). The data grid is a fifth consumer. Any formula tweak must be hand-mirrored across all of them. This is the single clearest embodiment of the "scoring in multiple places" problem.

4.2 Era and park are applied at inconsistent layers. For the basic scores, era and park are applied inside the core, so the stored basic columns already include them. For the advanced wOBA scores, the core produces a raw value with no era or park, and they are applied later, in the consumer. So whether a stored column already contains era and park depends on the metric, and anyone consuming a column has to know which regime it is in. This is the "what does each step accomplish, and where" ambiguity, made concrete.

4.3 The anchor and scale model. A hard-coded wOBA target anchors the pool: per-event scales bring the top-N anchor pool to baseline rates, then a per-side scale brings the anchor pool's mean recomputed wOBA to the target. Anchors are selected from the calibration pool rather than the full collection (which the code comment confirms is correct, since anchoring against the full collection collapses the scales). Hitters get per-side scales; pitchers get a single scale applied to both sides.

4.4 Cross-pool hitter/pitcher comparability is a power transform, and display does not equal optimizer. The optimizer objective raises each calibrated score to a fixed power and multiplies by a normalizer so hitter and pitcher coefficients sit on comparable scales, and this is explicitly optimizer-only. A power transform is monotonic, so ordering within a pool is preserved, but the cross-pool hitter-versus-pitcher tradeoffs the optimizer makes are not reflected in any number the user sees. A deeper read found two cross-pool mechanisms coexisting (a linear multiplier that is inert outside cap mode, and the global power transform), applied inconsistently across the display paths, plus three separately-defined power exponents that appear intended to be equal but could drift. This directly informed the later decision (D2) to drop the power transform and put both pools on one honest scale.

4.5 What crosses the wire (the boundary that should not exist). Because the frontend sends the model coefficients themselves to the backend, the backend can and does re-run the scoring math. In a single-core design none of that bundle needs to exist: the core would emit final consumable values and the optimizer would read them.

4.6 A fourth persistence mechanism. The roster page adds sessionStorage for the working roster on top of IndexedDB, localStorage, and the calibration cache. Four storage layers in total, which is what the persistence decision (D7) later consolidates.

4.7 Implication for the rebuild. A single scoring core must own the per-event model, the BIP recompute (once, in one place), era and park, calibration and anchoring, and cross-pool normalization, and emit one final score per card, side, and role that the grid, roster page, optimizer, and SP all read without recomputation. The genuinely open questions that remain: the correct order of calibration versus era and park; whether per-event scaling, anchor scalar, and power-norm are three necessary steps or collapsible into fewer; and whether cross-pool comparability should be a power transform, a shared unit, or something the optimizer handles directly.

5. Why the post-processing phases exist (root-cause analysis)

This is the analysis that most changed the rebuild plan. The optimizer does not build the full 26-man roster. It optimizes the high-value core (starters and rotation) and deliberately reserves cap budget for support roles, which are then filled by post-LP greedy passes within that reserved budget. Almost every post-processing phase is a consequence of that one choice:

  • Bench fill exists only because the optimizer did not select the bench. It is the home of a known greedy-versus-optimal value bug.
  • The backup-catcher three-pass exists because "a starter plus a qualified backup catcher" could not be guaranteed in the core LP, so it runs three full re-solves. It is the single most workaround-heavy mechanism.
  • Cap reclaim exists purely because the reserved support budget may go unspent; it reclaims the surplus and re-solves to upgrade starters.
  • Bonus selection is a separate concern: a two-way player fills two roles with one card, freeing a slot to fill.

The conclusion: bench fill, the backup-catcher three-pass, cap reclaim, and swingman/reliever fill are all consequences of "optimize starters only and reserve a support sub-budget," not fundamental inexpressibility. A single LP selecting all 26 with full coverage (including a backup-qualified catcher), rotation and bullpen slots, and the whole cap with no reservation would collapse them into the solve. This is exactly what fed decision D5: keep the decomposition where it earns its keep, but replace the greedy fill with a principled matching-based approach and make defensive feasibility a real constraint rather than a three-pass workaround.

6. Model training and the functional-form question

A read of the training code confirms the handover's description here is accurate and pins the form. The fit is strictly log-linear, a single natural-log term per event, by weighted least squares; no quadratic or cubic terms are ever fit, so the extra coefficient slots are dead. Weighting is by plate appearances (or batters faced) to the 0.75 power. Training matches inference: the model uses predicted BIP from the just-fit sub-models, mirroring the scoring-time recompute. Left and right splits are kept as separate observations feeding one unified fit. The implication: the functional-form concern is real and untouched, because the model genuinely is single-log, and the residual shape (over-predict the middle, under-predict the high end) is a property of that form. The rebuild's "fit and compare forms" task is wide open; nothing in the code pre-judges it.

There is also prior art. A separate root-level Python script fits a sequential conditional cubic-logistic model (each event conditioned on the prior ones, so the plate-appearance accounting is enforced by structure). It is not what the app trains, but it explains the otherwise-orphaned cubic slots, and it is a serious alternative worth comparing against the single-log form in the bake-off.

7. Remaining surfaces

Variants and base edits. Variants are separate appended rows derived from base rows by a fixed boost formula, one per card, persisted and re-derivable after re-import. Base inline edits, by contrast, are ephemeral: they write to the working copy and are wiped on the next import. That matters for the ownership-override use case and is a behavior to decide on deliberately in the rebuild.

Config flow, and the real correctness trap. Scoring always reads the coefficients store. On a tournament switch, only the position weights are auto-pushed into it; era, park, and softcaps enter the store only when the user clicks a "load into coefficients" button. So selecting a tournament does not make scoring use that tournament's run environment until a manual button press. This is a genuine correctness trap, and it is the concrete reason the config decision (D4) makes tournament selection directly drive the scoring config from one source. Config is duplicated across three homes that must agree (the eras library, the tournament, and the coefficients store), which is what that decision collapses.

The data grid scores through the core and exposes the extensibility hooks (column presets, external filters, a dataset override) that make it reusable for Single Player. Two handover "wants," named column presets and a guard against corrupt rating values, were confirmed absent rather than existing.

A latent bug found in passing. The same pitch-type-counting logic lives in two places with two different column-name spellings; one uses the correct no-space names, the other looks for names with spaces that never match, so two pitch types are never counted, affecting starter qualification. A small, concrete instance of the same "one concept, two definitions" pattern the whole discovery is about.

PT Account (the one net-new feature). Roster generation keeps an owned/not-owned option, scoped per active account rather than pooling across accounts; one CSV import per account; ownership is a quantity already in the export, so "owned" means quantity greater than zero. It applies to Perfect Team only. The natural data shape is one shared card catalog plus a per-account overlay carrying ownership and that account's variants, so importing two accounts reconciles to one catalog with two overlays rather than two duplicate datasets.

8. Coverage

Every non-vendored source file was read or structurally characterized. The only items intentionally not deep-read are pure UI-entry widgets whose function is fully implied by their usage. Nothing scoring-, optimizer-, config-, or persistence-relevant remains unexamined.
