Productivity App: v2 Scope and Prioritization

Work sample · v2 scope and prioritization · as of v1 (July 2026)

What this is: a menu of candidate v2 scope, each option recording how v1 classified it, its current state in the shipped app, and what building it would involve, so it can be prioritized.

What it shows: prioritization, deliberate scoping, and forward roadmapping past a shipped v1.

Source: lightly edited from the project's v2 scope-options document for presentation (internal references cleaned). The full working version lives in the app repo.

This is a planning menu, not a commitment or a plan. It compiles everything the v1 docs deferred so a next round of work can be prioritized. v1 (Phases 1 to 3) is complete.

How this document is organized

Every option carries an Origin tag showing how the v1 docs classified it, because the distinction matters for prioritization: "Out of scope" items were set aside deliberately, while "Future" items were always intended to come back.

Origin tags:

  • Out of scope (v1): listed under "Out of Scope for v1" in the PRD. Deliberately excluded from v1, with no commitment to a version.
  • Future Direction: labeled "Future Direction" in the PRD or technical architecture spec. Intended for a later version, not v1.
  • v1.1: explicitly slated for v1.1 in the docs.

A few options carry more than one tag (for example, duration goals are both "Out of scope (v1)" and "Future Direction"). Where that is the case, all applicable tags are shown.

Current-state tags:

  • Not started: no code exists for it.
  • Partial (foundation): a dependency or data layer it needs already ships in v1, but the feature itself is not built.

Deferred features

# Option Origin Current state What v2 would involve
1 Pattern analytics (trends, streaks, consistency over time) Out of scope (v1); analytics views flagged v1.1 Partial (foundation): every completion is stored, and per-item history shows in item detail, but there are no trend, streak, or aggregate views New read-only surfaces over existing completion data (charts, streaks, per-tag or per-period rollups). No new data model.
2 Notifications and reminders Out of scope (v1); server-scheduling need flagged v1.1 Not started The first scheduled/server work in the app (the current architecture has no scheduled jobs). Needs push infrastructure and per-item reminder settings.
3 Hour-based (duration) habit goals (e.g. "2 hours per week") Out of scope (v1) and Future Direction Not started (deliberately: goals are count/frequency only; no duration-target field in the schema) A second habit-goal kind alongside count: a duration target, logging minutes per completion, and duration progress on the Progress surface. Schema and progress-math changes.
4 Natural language input (type "run 7am tomorrow") Out of scope (v1) Not started A parser that turns free text into the item fields (date, time, repeat). Sits on top of the existing item model.
5 AI-assisted scheduling (suggest when to do things) Out of scope (v1) Not started Suggestions layered over the day/week; needs a suggestion engine and UI. Largest and least-defined option.
6 Data export (take your data out) Out of scope (v1) Not started Export items, completions, and tags to a portable format (JSON/CSV/ICS). Reads the local store; no new data.
7 Additional calendar integrations (Apple Calendar, Outlook) Out of scope (v1) Not started Reuse the Google Calendar sync pattern for other providers. Each is its own OAuth + API integration.
8 Optional gamification (streaks as rewards, badges) Out of scope (v1) Not started A motivational layer over completion data. Overlaps with pattern analytics (streaks).
9 First-run onboarding flow Out of scope (v1) Not started (by design: v1 is "no setup required" and opens straight to Plan) An optional guided intro. Tension with the "zero configuration" principle, so worth a design decision on whether it is wanted at all.
10 Mobile application Out of scope (v1) Partial (foundation): the app is an installable PWA, but the UI is desktop-first and there is no mobile-optimized layout Either a responsive mobile layout pass on the existing PWA, or a native app. The narrow-screen Week/Month fallback (see refinements below) is the concrete first step.
11 Social / collaborative features (sharing, multi-user) Out of scope (v1) Not started (single-user by design; conflict resolution is last-write-wins, not collaborative) The largest architectural change: multi-user data, sharing, and real conflict handling. Contradicts the current single-user design premise.

Tracked deferred story (already in the backlog)

Unlike the items above (which live only as prose in the v1 docs), this one is a groomed Jira story:

  • Google Drive file attachments in item notes (PRD-81). Origin: split off from the rich-text notes work (was Phase 2, relabeled Phase 3), not part of the original "Out of scope" list. Current state: Not started, held. Groomed as the Google Drive Picker approach (Option B), blocked on Google Cloud setup. The Google OAuth foundation it builds on already exists. The full setup checklist and two-phase build plan live in the PRD-81 ticket.

Deferred design and polish (behavior shipped, refinement left open)

These are not new features; they are refinements the v1 docs flagged as "still open for later phases." The underlying behavior works today.

  • Narrow-screen, mobile-responsive layout for the Week and Month grids. PRD-91 shipped these desktop-first and explicitly deferred the narrow-screen fallback. This is the concrete first step toward option 10 (mobile).
  • Guest-to-account merge prompt presentation and duplicate cleanup. The merge behavior is built and correct (tag reconciliation was fixed under PRD-103). How the prompt and any duplicate cleanup are presented was left for refinement in the information architecture spec.

Resolved since these notes were written (no longer future)

Recorded so this list is not mistaken for still-open work. The v1 docs once listed these as "later phase," and they are now shipped:

  • Category lens at the Week and Month ranges (PRD-102).
  • The mini-calendar date navigator (PRD-56).
  • How tags display in item rows and the Category lens (Phase 2).

Notes for prioritizing

  • Cheapest, highest-leverage on existing data: pattern analytics (1) and data export (6) both read the completion/item data that already ships, with no new data model.
  • Completes an existing feature: duration habit goals (3) extend the Progress surface that already exists for count goals.
  • Biggest lift / most design-dependent: AI-assisted scheduling (5), social/collaborative (11), and a native mobile app (10, if native rather than responsive).
  • Needs external setup before any build: additional calendar integrations (7) and the tracked Drive-attachments story (PRD-81) each require provider/OAuth setup.
