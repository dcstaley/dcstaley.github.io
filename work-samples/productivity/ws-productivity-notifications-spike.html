Notifications and Reminders: Feature Spike and Decision Doc

Work sample · feature spike and decision doc · as of v1 (July 2026)

What this is: a spike and decision document for a proposed v2 feature (notifications and reminders), grounding the design in the real codebase and asking for a go/no-go on the first phase.

What it shows: feature discovery and de-risking, plus a real decision-forcing "green-light Phase A" ask, the mechanics of product decision-making.

Source: condensed and edited from the project's notifications spike for presentation (trimmed for length, internal references removed). The full working version lives in the app repo.

The app is a local-first, guest-usable tasks-and-habits web app: it runs entirely on-device, and an account is optional. This spike asks a hard question up front: can we deliver reminders that fire when the app is closed, and what does that cost us in architecture and money? It answers by grounding the design in how the app already works, then asks to green-light an infrastructure-only first phase.

1. Goal and user-facing behavior

Reminders tell the user about a thing to do at (or before) the moment it matters, even when the app isn't open. Three concrete jobs, mapped to the item shapes the app already has:

  • Timed-item reminders. For an item with a start and end time, fire a notification a chosen amount before the start ("Standup in 10 minutes"). The offset is per-item or a global default.
  • All-day and by-end-of-day reminders. For items with a date but no clock time, there is no start to count back from, so these fire at a fixed daily clock time the user picks ("Today: Pay rent"). The same applies to open-window items, anchored to the window start.
  • Habit nudges. For a tracked habit due today with no completion recorded yet, nudge later in the day ("You haven't done Meditate today"). This reuses the app's existing due-but-not-done derivation rather than inventing new logic.

Tapping a notification deep-links into the app on the right day. Delivered notifications are transient: nothing about them is stored as app data. Reminders are a read-only layer on top of the item model; they must never change completion or scheduling data.

2. The core constraint

This app has no server-scheduled jobs today, and real reminders require them. The architecture is explicit that nothing runs on a schedule, and the one named future exception is exactly this feature. The whole app is local-first; the only server code is the request-driven Google Calendar routes.

A reminder's defining requirement is that it fires when the app is closed. Every option that stays inside the current architecture fails that test:

  • A page timer only runs while a tab is open.
  • Service-worker timers don't survive. The service worker is started for an event and terminated seconds later. The timestamp-based scheduling API that would work around this is experimental, Chromium-only, and absent on iOS.
  • Periodic background sync is Chromium-only, throttled to roughly daily, install-gated, and absent on iOS.

The service worker is essential only as the receiver: it handles an incoming push and shows the notification. Something off-device has to decide "it's time" and wake the device. That forces two pieces the app lacks: Web Push (a per-device subscription plus a service-worker push handler) and a server-side scheduler (a job on a clock that finds due items and sends the pushes).

3. Recommended architecture on this stack

Add Web Push delivery plus a scheduled server function, slotting into the existing structure without disturbing the local-first core. The moving parts:

  • VAPID keys, the keypair that authorizes Web Push. The public key ships to the client; the private key stays server-only, in environment variables.
  • A push handler in the service worker that shows the notification and deep-links on tap.
  • Subscription capture: ask permission, subscribe the device, and POST the subscription to a server route. This lives in a small client-side controller in the app shell, beside the existing calendar-sync controller.
  • A push_subscriptions table holding each device's endpoint and keys, keyed by user. It is server-only, the same way calendar connections are deliberately kept out of the synced local store.
  • A subscription API to save and remove subscriptions, authenticated the same way the calendar routes are.
  • The scheduler, a cron entry that hits a route every few minutes. This is the first scheduled job in the app.
  • The send function, which reads due items server-side, computes who is due now in their own timezone, and sends a push per subscription.
  • Reminder settings, new columns describing whether and when to remind. These are ordinary app data and do sync.

Flow: enabling asks permission, subscribes the device, and saves the subscription. Delivery runs on the clock: the scheduled job reads due items and each user's timezone server-side, sends a push per subscription, and drops dead endpoints. Receiving wakes the service worker, which shows the notification; a tap deep-links to the right day.

The key point: the send function reads the same server tables the local store already syncs to, so the scheduled job has everything it needs server-side with no new sync path.

Precedent to mirror: the existing calendar outbox already does the hard version of this. It is a device-local controller in the app shell that sends to the server without blocking the UI, records a structured outcome per item, and retries. Reminders reuse that pattern. The one difference is that subscription capture goes client to server, because the device owns the subscription.

4. Data model additions

Two kinds of new data sit on opposite sides of the sync boundary, mirroring how calendar connections are deliberately excluded from the synced store.

A. Reminder settings on items (and defaults on the profile). These do sync. Add a reminder-enabled flag, an offset in minutes (for timed items), and a clock time (for all-day, end-of-day, open-window, and nudge cases). These go into the local schema, a server migration, and the validation schema together, keeping the schema-sync discipline the project already follows. A global default (default offset, default time, quiet hours) sits beside the existing timezone setting. Because these are ordinary item columns, they sync to the server, which is what puts them in the scheduled job's reach.

B. push_subscriptions. Server-only, not synced. The direct analogue of calendar connections: an id, the user id (the access-control key), the endpoint, the two subscription keys, and a created-at timestamp. Row-level security scopes the client's own subscribe and unsubscribe to its user; the scheduled send uses a service-role client with no user session, the same reasoning as the OAuth callback. The VAPID private key and the service-role key never leave the server.

5. Implications and limits

  • Account-only, and this is inherent, not a temporary gap. Web Push needs a subscription tied to a user and a server that can read that user's items. A guest keeps data only on-device, so there is nothing server-side to read and no user to key a subscription to. Closed-app reminders are therefore a signed-in feature, gated the same way Google Calendar sync is.
  • Offline. Subscribing needs the network; after that, delivery is server-driven, so an offline device just receives the push when its push service is next reachable. The subscribe call should tolerate being offline and retry when back online.
  • iOS needs an installed PWA (16.4+), with quirks. Web Push on iPhone and iPad works only for the home-screen-installed app on 16.4 and later, and the permission prompt must follow a user gesture. The app is already installable, but the settings UI has to detect "not installed or unsupported" and explain the Add-to-Home-Screen step instead of failing silently. Desktop and Android Chrome are more forgiving.
  • Permission UX. Never prompt on load. Request permission only when the user turns reminders on, from a Settings toggle beside the calendar panel. A denied permission gets an honest recovery note.
  • Timezone correctness is non-negotiable and already solvable. The app stores each user's timezone and anchors today, overdue, and missed to it. The scheduled run executes in UTC, so it has to compute "due now?" in the user's timezone. That logic should be a pure, unit-tested module, consistent with the existing recurrence and day-query code.

6. Phased build plan

Phase A: infrastructure plus one manual test round-trip. Prove the whole pipe with a single button before any per-item logic. Generate the VAPID keys (the owner pastes them into the environment; no secret is handled in the build session), add the push and tap handlers to the service worker, create the subscriptions table with row-level security, add the subscribe and unsubscribe routes, add the client controller and a minimal Settings toggle that requests permission and subscribes, and add a manual "send me a test" route. Verify by installing the PWA, toggling on, clicking Send test, and seeing a notification, first with the app open and then with it closed. No scheduler yet.

Phase B: per-item timed reminders, settings, and permission flow. Add the reminder columns to items and the profile (schema, migration, and validation) and controls to the item form; write the pure, timezone-aware "due for a reminder in this window?" module with tests; add the cron entry and the run route (service-role read plus the push send, dropping dead endpoints); handle the not-installed, unsupported, and denied states. Verify by setting a reminder a few minutes out, closing the app, receiving it, and tapping through to the right day.

Phase C: habit nudges. Extend the reminders module to derive "due today but not completed" (reusing the existing recurrence logic and the completion check), unit-tested; add a per-habit nudge time and copy. Verify that a due, undone habit nudges at the set time, and that completing it beforehand suppresses the nudge.

7. Risks, cost, and open questions

The scheduler cost question (and the recommended answer). Reminders that fire near a chosen minute need a run every 5 to 15 minutes. The hosting platform's free cron tier is limited to roughly daily granularity, which would push the platform's cron toward a paid plan. The recommended path avoids that: run the trigger on the database we already pay nothing for (Supabase pg_cron via pg_net) on a roughly 5-minute schedule, and keep the push send in a Node route on the existing host. Because reminders fire minutes-before, 5-minute granularity is fine, and this keeps the feature at no recurring cost. The alternative (the host's paid cron) stays on the table if the database trigger proves awkward.

Other risks:

  • Web Push is best-effort. Delivery can be delayed or dropped on battery-saver or backgrounded mobile. Treat reminders as nudges, not guarantees, and never assume exactly-once. Prune dead endpoints.
  • VAPID key management. The private key is a long-lived server secret; rotating it forces every device to re-subscribe.
  • iOS fragility (installed PWA, 16.4+, gesture-gated) is the highest-support surface.
  • Notification fatigue. Conservative defaults, quiet hours, and easy per-item opt-out matter more than adding features.

Open product questions:

  1. Default reminder offset for timed items (10 minutes? 15?) and default daily clock time for all-day and end-of-day items (9:00am?).
  2. Quiet hours: a global "don't notify between X and Y." Added scope, but it avoids a 6am nudge.
  3. Per-item versus global control: a global default with a per-item override (as sketched), or global-only first?
  4. Which items remind by default: off by default for every item, or on for some?

Outcome: Phase A was green-lit and is now epic PRD-106 in the v2 backlog, with Phases B and C as follow-on epics.
