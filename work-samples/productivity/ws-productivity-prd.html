Product Requirements Document: Unified Tasks & Habits App

Work sample · product requirements document · as of v1 (July 2026)

What this is: the PRD for my unified tasks-and-habits web app, defining what to build and how it should behave across three phases.

What it shows: requirements definition and scope discipline (including an explicit out-of-scope list), the product decisions behind the app, and a version changelog that records how the spec evolved, with several deliberate reversals.

Source: lightly edited from the project's PRD for presentation (internal references cleaned, some implementation-level detail trimmed). The full working version lives in the app repo.

This doc tracked the product from initial draft through shipped v1. It is written in planning tense because that is what it is: the spec the build worked from. The changelog at the end is where the decision history lives.

Overview

A unified productivity web application for managing tasks and habits in a single system. Single user, no collaboration features. The primary user opens it daily to plan their day, track recurring habits, and manage personal commitments alongside an existing work calendar.

The central concept is flexible scheduling granularity: items are not forced into a specific time slot or a flat daily list. They can exist at different levels of scheduling specificity and coexist in the same view.

Scope

In scope for v1

  • Phase 1 (Core Experience): local-first foundation (works offline), guest (no-account) use, optional accounts with sync and backup, item CRUD (with optional repeat and optional habit tracking), the calendar day grid plus rail, flexible scheduling, completion marking. No setup required.
  • Phase 2 (Integration and Configuration): completion history tracking, notes field, Google Calendar sync, tags and the by-tag Category view.
  • Phase 3 (Views and Quantitative Tracking): additional time ranges (Week and Month) and lenses (Type: Habits, Tasks, and Events), configurable landing view, quantitative progress tracking.

Out of scope for v1

Explicitly deferred, and not to influence v1 implementation decisions: a mobile application; notifications and reminders; pattern analytics (the data layer ships in v1, analytics views are v1.1); natural language input; AI-assisted scheduling; data export; optional gamification; hour-based (duration) habit goals (count and frequency goals are in v1, duration is deferred); additional calendar integrations (Apple, Outlook); social or collaborative features; and a first-run onboarding flow.

Key decisions

Local-first, offline, and guest use. The app works fully offline and can be used without an account; data lives on the device first. An account is optional and exists to enable sync, backup, and multi-device, not to gate access. Offline and signed-out are distinct: a signed-in user offline keeps their session and full local access, and only an explicit sign-out clears local data and returns the device to guest. On first sign-up, a guest's local data migrates into the account; signing into an account that already has data prompts once to add the local data (default) or discard it, reconciling same-named tags rather than duplicating them. Cross-device conflict resolution is last-write-wins. Google Calendar sync requires an account and a connection and works only online.

Authentication. Optional; it turns on sync and backup rather than gating access. Two methods, Google OAuth and email/password, are independent paths that cannot merge or switch after creation. Each account maps to one set of app data, and sessions persist across browser closes.

Google Calendar sync. Push-only by default: time-specific and all-day items sync to a dedicated app calendar. External events are not pulled into the app unless bidirectional sync is enabled (an optional toggle). Google OAuth users need no second login; email/password users grant calendar access through a separate flow in settings that does not change their sign-in method.

Items, repeat, and habit tracking. One underlying item type; there is no task-versus-habit choice at creation. Repeat is one control with two flavors: a fixed schedule (every day, specific weekdays, every N days/weeks/months), available on any timing mode; or a frequency (N times per period, for example four times a week), available only on the flexible modes, since a set-time block is pinned to a clock time. Both can end (never, on a date, or after N). Repetition alone does not make something a habit; a weekly meeting just repeats. "Track as a habit" is an optional toggle that turns on consistency tracking against the item's recurrence, with no separate target to fill in (a count habit's target is its frequency, and a binary habit is a scheduled item with tracking on). "Task" and "habit" are shorthand: a habit is simply an item with tracking on.

Scheduling (the "when"). Set at creation as one of three timing modes: Set time (a date, or date range, with a start and end time, where the end is the duration); Timeframe (a date plus "between X and Y," optionally "for a duration," done any time in the range); or No set time (no date or time, a rail item, optionally recurring). Within Set time, two checkboxes: All day (occupies the whole day, shown in the all-day band, can span a date range) and By end of day (due that day, shown on the rail). This replaced an earlier flat list of scheduling types; an item is never created "as Morning."

Tags, one organizing layer. Tags are user-defined labels (Work, Fitness, Personal) used to group and filter items. They replace the former scheduling-group concept entirely; there are no Morning/Afternoon/Evening groups. An item can carry any number of tags; tags arrive in Phase 2.

No setup required. The app is usable immediately with zero configuration. The day grid works out of the box; tags are optional. This replaced the former "smart default groups."

View model. Two independent controls present the same items: a time range (Day, Week, Month) and an organize-by lens (Schedule, Category, Type). The Schedule lens shows the calendar surface (day grid plus rail at Day, a grid at Week and Month). The Category lens groups and filters by tag. The Type lens splits into Habits, Tasks, and Events. Day plus Schedule is the Phase 1 daily planning view; the Category view arrives in Phase 2; the other ranges and the Type lens arrive in Phase 3. The landing view is user-configurable, defaulting to Day plus Schedule.

Overdue and incomplete states. Set-time items become overdue once their end time passes without completion (all-day and by-end-of-day once the day passes). Timeframe items become overdue once the window passes; the item is not deleted. No-set-time items are never "overdue"; a floating item just stays on the rail, and a frequency habit whose period closes with the count unmet is counted as missed, not overdue. These are derived display states, not stored fields: missed instances are computed on demand from the recurrence and the completion records, and repeat rules are effective-dated so a past date is always judged against the rule that applied then, never recomputed from a later version.

Phase 1: Core Experience

Goal: the app is usable for the first time. A person can open it (as guest or signed in), see their day on the calendar grid, create items, optionally repeat or track them as habits, and mark them complete.

Authentication. Optional, and never blocks use. A new user starts immediately as a guest, with all data stored locally and the app fully functional offline; a gentle, dismissible prompt suggests signing up so data is backed up. Both methods live on the same sign-in screen. Creating an account turns on sync and backup and migrates the guest's existing local data into the account so nothing is lost.

The guest-to-account handling is deliberately careful about the one genuinely ambiguous case. Signing up to a new (or empty) account claims and uploads the local data with no prompt; signing in on a fresh device syncs the account down with no prompt; only signing in where both the device and the account already hold data prompts once ("Add the items you created on this device to your account?", Add as the default). A merge there is a union of independently created records, not a last-write-wins conflict, since the records have separate IDs; to avoid obvious duplicates, same-named tags are reconciled case-insensitively and guest items repoint to the account's matching tag. One account maps to one email: signing in with Google for an email that already has a password account links the two rather than erroring.

Sign out is the only action that clears local data. It first confirms changes have synced (warning if unsynced offline changes would be lost), then clears the device and returns it to guest. Being offline is not sign-out. Google Calendar access is not revoked on sign-out; the user disconnects it explicitly.

Creating items. There is one kind of item. To create one, the user gives it a title and a timing; repeat and habit-tracking are optional layers on top. The form stays simple: a small three-option mode selector plus a couple of checkboxes.

  • Set time (default): a date or date range with a start and end time, shown as a block on the day grid. Eligible for Google Calendar sync. Two checkboxes grey out the time: All day (spans whole days, shown in the all-day band) and By end of day (due that day, shown on the rail).
  • Timeframe: a date plus "do it between X and Y," optionally "for a duration." Flexible within the range; lives on the rail, with its window shown as a focus-band when selected.
  • No set time: no date or time. Combined with a frequency it expresses "Practice Spanish four times a week"; with no recurrence it is a floating to-do.

Repeat is one control with two flavors (a schedule, or a frequency of N per period) plus an end. "Track as a habit" turns on consistency tracking with no separate target. Repeating and habit-tracking are independent:

Item Timing Repeat Tracked?
Meeting with contractor Set time, 1–2 PM No No
Vacation in Mexico Set time, All day, Jun 18–25 No No
Call Mom Set time, By end of day No No
Play guitar Timeframe, 12–4 PM, for 1 hr No No
Team standup, 9:00 AM Set time Every weekday No
Meditate Set time or no set time Every day Yes (binary)
Practice Spanish No set time 4 times per week Yes (count = the frequency)

Editing and deleting. All fields are editable; editing a repeat rule affects future instances only, since rules are effective-dated. Deleting shows a confirmation. Deleting a habit soft-deletes it (the item and its completion history are retained but hidden, so history is preserved); deleting a task or event removes it and its completion records permanently. This split applies from Phase 1, since completions exist from Phase 1.

Daily planning view. The primary surface, and the Day range of the Schedule lens: a calendar day grid showing the current day, in three zones. An all-day band on top; a time grid down the day where set-time items appear as blocks; and a side rail holding everything without a set time (timeframe, by-end-of-day, and no-set-time items active today, including tracked habits due today). The user can drag a rail item onto the grid to give it a time. Only what is relevant to the day appears; future-only items, inactive repeats, and completed items from previous days do not. Day navigation moves forward and backward: past days are viewable and their completion state is editable (a forgotten item can be marked done), and future days show what is scheduled without a completion control.

Because everything is one item type, there are no separate "tasks" and "habits" tabs. Tracked habits and ordinary items appear together, visually distinguished rather than split into modules. This unified model is a core differentiator.

Completion marking. In Phase 1, completion is a simple per-item, per-day toggle, reversible within the day. A count habit uses a progress ring instead of a checkbox: tapping it logs a completion and fills the ring toward the target, a teal check shows at the target, and once met a separate "Log again" control allows logging past the goal (for example 4 of 3 this week). The count habit stays on the rail for its whole active period and shows plain-text period progress ("2 of 4"). Recurring items reset each day; non-recurring tasks stay complete once toggled. Completion in Phase 1 does not yet surface history, streaks, or rates (those read the same records in Phase 2). The completion data store is provisioned from Phase 1 so completion state is durable from the start, which keeps the behavior phasing intact while avoiding a later data migration.

Phase 2: Integration and Configuration

Goal: the app works alongside a user's existing calendar, tags organize items, and items carry notes and a persistent completion record.

Completion history. Phase 1 already stores each completion; Phase 2 adds the surfaces that read those records over time. When an item is marked done, a completion event is stored (item ID, date, timestamp); for a count habit, each completion is its own event. Missed instances are not stored: a tracked habit counts as missed for a date when it was active then (per the effective repeat rule) and has no completion record, computed on demand so no end-of-day process runs. Past daily views show what was scheduled and whether each item was completed or missed, and completion is editable for past dates.

Notes. A freeform, optional rich-text field on all items, matching Google Calendar's description toolbar (bold, italic, underline, lists, links) and stored as HTML so notes sync cleanly, rendered sanitized. Notes are visible in the item detail view; no preview in the daily list in v1.

Google Calendar sync. Connects the app to the user's Google Calendar so time-specific items appear alongside existing events. Requires an account, works only online, and queues offline changes until reconnected. Push-only by default: time-specific and all-day items sync to a dedicated app calendar (by-end-of-day, timeframe, and someday items do not map to a slot and are not synced). Editing or deleting a synced item propagates automatically by default, governed by an "Automatically sync event changes" toggle; turning it off makes the app ask per change. An optional bidirectional toggle (off by default) surfaces external events read-only on their date, inert and not editable. Sync never blocks item creation or editing; failures show a non-blocking status.

Tags and the Category view. User-defined labels that group and filter items, the product's single organizing layer. Users create, rename, and delete tags from settings, with rename and delete also available in-flow on the Category lens section headers. Deleting a tag removes it from items without deleting the items. The Category view gives each tag a section with its items beneath, items with multiple tags appear under each, and an "Untagged" section collects the rest.

Phase 3: Views and Quantitative Tracking

Goal: users can view items across additional time ranges and lenses, and log progress against quantitative targets.

Views. The same items seen through two independent controls. The time range is Day, Week, or Month. The lens is Schedule (the calendar surface: day grid plus rail at Day, a calendar grid at Week and Month), Category (grouped by tag), or Type (three sections, Habits, Tasks, and Events, where Events are set-time items, Tasks are timeframe and no-set-time, and Habits are the tracked habits). The two controls are independent, so any combination is valid ("this month, by category" or "today, by type"). The landing view is a user setting, defaulting to Day plus Schedule.

The Week and Month Schedule grids keep one shell at every zoom: a calendar area on the left that changes with the range, and a persistent Tasks panel on the right (the Phase 1 rail, now present at all three ranges). Week and Month reuse the Day view's per-item placement rule mapped across the span, so no new placement logic is introduced. Week is a seven-day header plus an all-day band plus a scrollable time grid; Month is a five-to-six-week grid of compact date cells with up to three item chips and a "+N more" affordance. The Tasks panel groups items into Recurring (repeating rail items shown once for the span), Anytime (undated one-offs, also shown once), and per-day groups. First day of week is configurable. External Google events render read-only and visually distinct, never in the Tasks panel. Desktop-first, with a narrow-screen fallback deferred as a later refinement.

Quantitative progress. A dedicated surface for viewing progress against frequency (count) goals, reading the same completion records used everywhere. It lists tracked habits with a frequency goal and shows, per habit, the goal, progress in the current period, days remaining, and a quiet progress indicator. Frequency periods use a hybrid model: a named calendar period (day, week, month) snaps to calendar boundaries, while a custom "N times per X days" window runs in consecutive windows anchored to the rule's start date. Nothing evaluates a trailing window from today. Logging is just marking the item done (plus "Log again" for multiple completions in a day); there is no separate duration entry in v1.

Settings

A persistent surface accessible from anywhere, whose contents grow across phases. Phase 1: account (a sign-up entry point for guests, or account display and logout for signed-in users), sync and offline status, and timezone (the timezone used to evaluate the day boundary, defaulting to the browser's). Phase 2: Google Calendar connection, the bidirectional-sync toggle, the automatic-change-sync toggle, and tag management. Phase 3: the default landing view.

Changelog

The spec ran from an initial draft to a shipped, complete v1. Entries are grouped by working date, each recording the decisions and reversals made that day (pure formatting and errata passes omitted).

v1.9, 2026-07-03: reconciled the doc with the shipped app.

  • Type lens became three sections (Habits, Tasks, Events), not "tracked habits versus the rest."
  • Count-habit control became a progress ring with a "Log again" past the goal.
  • Week/Month Tasks panel gained a "Recurring" group; tag rename and delete added to the Category lens headers.
  • Inbound read-only events widened to every visible Google calendar, not just the primary.

v1.8, 2026-07-02: Week/Month grid and Type lens.

  • Resolved the Week/Month Schedule grid, keeping the Tasks panel present at Day, Week, and Month (correcting an initial proposal that dropped it).
  • Added the Type lens at Day; fixed the completion model so a count habit is "done" only when its period target is met.

v1.7, 2026-07-01: Google Calendar sync shipped.

  • All-day items sync alongside time-specific ones; edits and deletes propagate automatically by default, governed by a new toggle.
  • Push is queued in a device-local outbox, so creation is never blocked on sync.
  • Added optional bidirectional sync, completing the Google Calendar epic and Phase 2.

v1.6, 2026-06-29: Notes upgraded to rich text.

  • Matched Google Calendar's toolbar, stored as HTML; split the Google Drive attachment into its own deferred story.

v1.5, 2026-06-24: account-conflict behavior changed to linking.

  • Signing in with Google for an email that already has a password account now links the two rather than erroring (one account per email, usable either way).

v1.4, 2026-06-23: count habit stays on the rail.

  • It stays for its whole active period rather than dropping off once the goal is met, so the user can keep logging past the target.

v1.3, 2026-06-22: past-date completion and delete behavior.

  • Made completion editable for past dates, reversing the earlier "past is read-only" stance, which was user-hostile to anyone who simply forgot.
  • Resolved delete behavior as a split by type from Phase 1 (habits soft-delete to preserve history; tasks and events delete permanently), replacing earlier wording that conflicted across sections.

v1.2, 2026-06-20: repeat model clarified.

  • A frequency repeat is available only on flexible timing modes; its period is hybrid (named calendar periods, or a custom anchored window).
  • Added a timezone setting for the day boundary.

v1.1, 2026-06-18: design-review pass, major model rework.

  • Unified to one item type with optional repeat and a "track as a habit" toggle, so recurrence no longer means habit; removed scheduling groups in favor of tags as the single organizing layer.
  • Reworked scheduling into three timing modes plus All-day and By-end-of-day checkboxes; dropped hour-based goals from v1 (Phase 3 trimmed to count-only).
  • Reworked the views model from four fixed views into two independent controls (a time range and an organize-by lens), with a configurable landing view.
  • Added offline and guest use as v1 requirements, making authentication optional.
  • Distinguished offline from signed-out, with sign-out the only action that clears local data.
  • Specified guest-to-account data handling across the four sign-up/sign-in cases (the both-populated case prompts once; a merge is a union of distinct records, not a conflict).
  • Made missed instances derived on demand rather than written by an end-of-day process, with effective-dated repeat rules.

v1.0, 2026-05-01: first complete spec.

  • Phase 3 specs written; authentication expanded to Google OAuth plus email/password; settings section added.

v0.1, 2026-04-30: initial draft.

  • Authentication, data model, scheduling, recurrence, and Phase 1 feature specs.
