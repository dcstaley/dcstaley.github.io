User Stories & Acceptance Criteria: Unified Tasks & Habits App

Work sample · user stories and acceptance criteria · as of v1 (July 2026)

What this is: the ticket-by-ticket build backlog for the app, turning the PRD's behavior into small, testable stories with acceptance criteria.

What it shows: how I decompose a product into vertical slices, write acceptance criteria (a hybrid of Given/When/Then scenarios and flat checklists), and hold a global Definition of Done, including speccing the hard edge cases rather than deferring them.

Source: condensed and edited from the project's user-stories document for presentation. The full backlog is about fifty tickets across the epics below; this exhibit keeps the methodology and Definition of Done in full and shows a representative sample of the tickets. The full working version lives in the app repo.

Derived from the PRD (behavior), the information architecture (screens), and the technical architecture (implementation): the PRD owns behavior, the technical architecture owns implementation, and this document turns both into ordered, testable tickets.

How to read this document

Each story is a vertical slice: a small, end-to-end piece of working behavior sized to roughly one build session and one pull request. Stories are grouped into epics, and the epics are ordered in the sequence they should be built, so working top to bottom gives a sensible build path. Dependencies are noted on each story where they matter.

Story shape. Every story has an ID, a user-story sentence ("As a / I want / so that"), and acceptance criteria. The ID prefix marks the epic (for example ITM-1), so tickets can be referenced in commits.

Acceptance criteria are hybrid. Behavior that depends on state or a trigger is written as Given / When / Then scenarios, because those map cleanly to implementation and to tests. Simple, flat rules (a field is required, a default value) are written as a checklist. A story uses whichever fits each criterion, so most stories have both.

Edge cases are included. Where a behavior has important edges (offline, guest, recurrence edits, derived "missed," the guest-to-account merge), those are written as their own scenarios rather than deferred, because they are where the logic actually has to be correct.

"User" means the single end user. This is a single-user app with no roles, so stories are written against a generic "user."

Phase scope. This document specs Phase 1 in full. Phase 2 and Phase 3 are captured as epic-level stubs at the end, to be expanded closer to their build, since their details will still move.

Global Definition of Done

These apply to every story in addition to its own acceptance criteria. A story is not done unless all of these hold. They are stated once here rather than repeated on each ticket.

  • The behavior works against the on-device local store and is instant; nothing blocks on the network (per the local-first architecture).
  • The behavior works for a guest with no account, unless the story is specifically about accounts, sync, or Google Calendar.
  • All user-facing reads exclude soft-deleted items (deleted_at IS NULL).
  • Input is validated client-side with Zod for UX; the schema and the inferred TypeScript types are the single source of types.
  • Any pure logic the story introduces (recurrence evaluation, the day query, missed/overdue derivation, progress math, the guest migration) has unit tests covering the happy path and the edge cases in the story.
  • The UI matches the Phase 1 mockups and design tokens (Inter; indigo accent #4F46E5; teal #0D9488 for the habit indicator; neutral grays; 8px corners), following the design system.
  • TypeScript is strict and the code passes ESLint and Prettier.
  • No em dashes appear in any UI copy, per the project writing convention.

Epic build order (Phase 1)

The epics build in this order; later epics depend on earlier ones.

  1. Foundation & Local Store (FND): the scaffold, the on-device store, the Supabase sync backend, the app shell, and the PWA. Nothing else can be built until the store and shell exist.
  2. Item Create & Edit (ITM): the create/edit overlay and all of item CRUD, so there is data to view and complete.
  3. Day / Plan View (PLAN): the daily planning surface that renders items from the store.
  4. Completion (DONE): marking items done, the toggle and "Log again," writing completion records.
  5. Accounts, Sync & Guest Migration (ACCT): optional accounts that turn on sync, and the guest-to-account data handling.
  6. Offline & PWA Behavior (OFF): the explicit offline behaviors and the sync/offline status surface.

Epics 1 through 4 deliver a fully usable guest app offline. Epics 5 and 6 layer accounts, sync, and the offline status surface on top without changing the core experience.

The stories (a representative sample)

The full backlog runs to about fifty tickets across the epics above. The six below are chosen to show the range: a foundation story, the core create flow, the habit differentiator, the count-habit completion behavior, the hardest data edge case, and a complex integration. The remaining tickets follow the same shape.

FND-2: On-device local store and schema

As a user, I want the app to keep its data on my device, so that it is instant and works with no account and no network.

Acceptance criteria

  • The on-device SQLite store is initialized via the local-first SDK.
  • The local schema mirrors the data model: profiles, items, completion_events (Phase 1 tables), with tags, item_tags, and calendar_connections provisioned for later phases.
  • The data layer exposes typed create / read / update / delete for items (delete is soft for habits, hard for non-habits) and read/write for completion_events, all against the local store.
  • Scenarios:
    • Given a fresh install with no account, When the app initializes, Then a local store exists and reads/writes succeed with sync turned off.
    • Given an item is written locally, When it is read back, Then the stored fields round-trip unchanged.

Checklist

  • All tables carry user_id, created_at, and (where defined) updated_at.
  • Date fields use ISO YYYY-MM-DD; timestamps use full ISO 8601.

ITM-1: Create a Set time item

As a user, I want to create an item with a date and a start and end time, so that it appears as a block on my day at the right time.

Acceptance criteria

  • Set time is the default mode when the create overlay opens.
  • Scenarios:
    • Given the overlay is in Set time mode, When I enter a title, a date, a start time, and an end time, and save, Then an item is created with scheduling_mode = set_time and it renders as a block on the grid at that time.
    • Given Set time mode, When I enter a start time but no end time, Then Save is disabled and an inline hint asks for an end time (the end is required because it is the duration).
    • Given Set time mode, When I enter an end time earlier than the start time, Then validation blocks save with a clear message.
    • Given no title, When I try to save, Then Save is disabled.

Checklist

  • date_start defaults to the day currently in view; is_all_day and is_end_of_day default to false.
  • Title is required, max 200 characters.

ITM-8: Track an item as a habit

As a user, I want a toggle to track an item as a habit, so that I can measure my consistency without creating a separate kind of item.

Acceptance criteria

  • Scenarios:
    • Given any item, When I turn on "Track as a habit" and save, Then is_tracked_habit is true and the item shows the habit indicator (teal bar) on the calendar.
    • Given a tracked item with a frequency repeat, When it is saved, Then it is a count habit whose target is its frequency.
    • Given a tracked item with a scheduled repeat, When it is saved, Then it is a binary habit (done / not done per active date).
    • Given the toggle, When I save, Then no separate numeric target is requested (hour-based goals are out of v1).

Checklist

  • The toggle is independent of repeat: an item can repeat without being tracked, and (rarely) be tracked while one-off.
  • The teal habit indicator appears on the calendar card, not on the item detail title.

DONE-3: Log again for a count habit

As a user, I want to log a count habit more than once a day, so that doing it twice counts twice toward my frequency goal.

Acceptance criteria

  • Scenarios:
    • Given a count habit (a tracked item with a frequency repeat) active that day, When it renders, Then its completion control is a progress ring: tapping it logs a completion and fills the ring toward the target, and a teal check shows once the target is met.
    • Given the target is met, When I tap the done ring again, Then that day's completions are cleared (an undo).
    • Given the target is met, When the day renders, Then a separate "Log again" control appears, and using it writes an additional completion for the same date, taking the period count past the target (it never clears).
    • Given the period target is met, When the day renders, Then the habit stays on the rail so the user can keep logging past the goal (for example 4 of 3 this week).
    • Given a binary or non-tracked item, When it renders, Then its control is a plain checkbox, with no ring, no "Log again," and no count indicator.

Checklist

  • "Log again" is a small secondary control at the trailing edge of the card, not a stepper, and never shifts the title.
  • Count progress in a period is the number of completion rows in that period; logging may exceed the target.

ACCT-10: Guest-to-account merge (both populated)

As a user signing into an account that already has data from a device that also has guest data, I want to choose whether to add my local items, so that I control the merge and avoid duplicates.

Acceptance criteria

  • Scenarios:
    • Given local guest data and an account that already has data, When I sign in, Then a one-time prompt asks "Add the items you created on this device to your account?" with Add as the default.
    • Given the prompt, When I choose Add, Then the two sets are unioned by identity: items and their completions upload as distinct rows, and same-named tags (case-insensitive) are reconciled so guest items repoint to the account's matching tag rather than duplicating it.
    • Given the prompt, When I choose discard, Then the local guest data is dropped after a clear warning, and only the account data remains.
    • Given an empty account or no local guest data, When I sign in, Then this prompt does not appear.

Checklist

  • The merge is a union of records with separate IDs, not a last-write-wins conflict.
  • Any genuine duplicates that remain can be deleted by the user.

GCAL-7: Optional bidirectional sync (read-only inbound events)

As a user, I want to optionally see my Google Calendar events inside the app, so that my day view reflects my whole schedule.

Acceptance criteria

  • Scenarios:
    • Given a connected user, When I enable bidirectional sync in Settings (off by default), Then my Google Calendar events appear on their date and time in the Day, Week, and Month ranges, distinguished as external.
    • Given I have several visible Google calendars, When inbound events render, Then events from every calendar I have visible in Google, plus the primary, are shown, excluding the app's own calendar (so pushed items never double up) and free/busy-only calendars; each event carries its source calendar's color as a small dot.
    • Given an external event in the app, When I try to edit it or mark it complete, Then I cannot (read-only; not a full app item, no repeat or tags).
    • Given bidirectional sync is off, Then no external events appear.

Checklist

  • Which calendars appear is controlled in Google (hide a calendar there and it stops appearing here); there is no in-app calendar chooser.
  • The toggle appears only once Calendar is connected, and defaults to off.

Phase 2 and Phase 3

Phase 2 layers integration and configuration on top of the core experience without changing Phase 1 behavior, across three epics: Tags & Category View (the single organizing layer), Notes & Completion History (a notes field plus the surfaces that read completion records over time), and Google Calendar Sync (the only feature with server-side code). Each is specced to full stories in the same shape as Phase 1.

Phase 3 is captured as epic stubs (Time Ranges, Type Lens, Progress, Configurable Landing View, and date/time display format), to be expanded to full stories when groomed, so the backlog shows the whole arc.
